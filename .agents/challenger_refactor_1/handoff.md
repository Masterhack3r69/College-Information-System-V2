## 2. Logic Chain
- Standard tests use H2 database with `ddl-auto: create-drop` and Flyway disabled (`application-test.yml`), which ignores the unique constraint `sections_unique_term` specified in Flyway files.
- Adding the constraint dynamically via JdbcTemplate in `SectionDuplicateCodeTests.setUp()` allows verification of the database constraint behavior.
- Calling `entityManager.flush()` triggers the insert operation immediately, raising a SQL unique constraint violation exception when duplicate keys exist.
- The assertion successfully caught the unique constraint violation exception, confirming database-level enforcement of unique section codes per term.
- Compilation and build execution of both Maven and npm build commands verify zero build-time syntax or type check issues.

## 3. Caveats
- The unique section code constraint relies strictly on the database schema constraint `sections_unique_term` generated by Flyway. It is not currently validated programmatically in `SectionService` before saving. Thus, any duplicate creation will result in a raw database transaction rollback or exception if not caught at the controller/exception-handler level.
